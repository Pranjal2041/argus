---
title: Session history
description: A durable record of every session that has run — name, node, and the folders it was in.
---

Sessions come and go. You spin one up on a cluster node, do some work, and kill it — and a week later
you cannot remember which machine it was on, or which folder it ran in. Session history fixes that: it
keeps a durable, per-host log of every session that has existed, so "where was that running?" always
has an answer.

## What it does

- **Records every session** it sees: its name, the node it ran on, and the folders it was in — a list
  per session, because a session's working directory can change as you `cd` around, each with
  timestamps.
- **Outlives the session.** The record stays after the session is killed, so you can recover where a
  now-gone session was running.
- **One merged view.** On macOS, open it with `⇧⌘Y` to see every session across all your machines,
  newest first, filterable by name, node, or folder. Folder paths are selectable so you can copy one
  straight out.

## How it works

The **broker** does the recording, not the app — it already polls its host a couple of times a second
to keep the session list fresh, so it folds each session it sees into a durable file on that host and
serves it at `GET /history`. Because every host keeps its own history, the app simply reads `/history`
from each broker and merges the results. A new session or a changed folder is written immediately, so
nothing is lost if a host restarts.

<Callout type="warn">
History is recorded going forward, from when a host first runs a broker version that supports it. It
is not retroactive, so it cannot recover a session that was already gone — but from then on,
everything is logged.
</Callout>
