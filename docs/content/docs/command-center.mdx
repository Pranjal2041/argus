The Command Center is the part of Argus that turns "a wall of terminals" into "a glance." Instead
of reading every session yourself, it shows one card per agent with a short, plain-English line
about what that agent is actually doing right now — and, more importantly, whether it **needs you**.

It is the answer to the question the whole project is built around: with agents running everywhere,
where should your attention go *first*?

## What it does

- **A card per agent**, across every machine, each with a one-line status that is *insight*, not a
  recap: the key result it just produced, the decision it is blocked on, the job it is monitoring,
  or where it got stuck.
- **A clear label** on each card — **Needs you**, **Stuck**, **Working**, **Worth a look**,
  **Milestone**, or **Idle** — so the few that want you stand out from the many that are fine.
- **Fixed sections**, top to bottom: *Needs you* first, then *Working*, then *Done & idle*, then a
  *Backlog* you set aside. Cards stay put as their text updates, so nothing flies around while you
  read.
- **Set a status yourself.** If a card is wrong, change it. Your correction is shown immediately and
  synced to your other devices, and the model is told you overrode it so it reasons about why rather
  than flipping it straight back.
- **Cross-device.** Your Mac does the work and publishes the result; your phone reads the same cards
  over the tailnet. Change a status on either and it propagates both ways.
- **Notifications.** When a session flips to needing you, Argus raises a notification and a Dock
  badge, so you do not have to keep watching — your attention is pulled to it only when it is earned.
  On macOS, an optional Caps Lock light alert can provide the same signal without sound.
- **Lab decisions share the same attention surface.** A pending experiment-set access request or a
  gated run appears in the top band with a flask mark and a fixed Lab label. Selecting it opens the
  exact evidence dossier in [Argus Lab](/docs/lab), rather than pretending the gate is a terminal
  status.

<Callout type="info">
It reads each agent's screen **passively** — nothing is ever typed into your agents to find out
what they are doing. Agent sessions started by the mesh (`ut spawn`) and hidden panels are left out
of the Command Center entirely.
</Callout>

## Caps Lock light alerts

On macOS, open **Settings → Attention** and enable **Blink Caps Lock light for Needs You**. The
default burst lasts 10 seconds and repeats every five minutes while at least one terminal agent or
Lab approval remains in the Command Center's top section. Both the burst duration and reminder
interval are configurable, and **Test Light** checks the currently connected keyboard immediately.

The independent **Blink when Working becomes Idle** option gives one shorter pulse (two seconds by
default) when a visible panel moves directly from Working to Idle. It does not repeat, and it stays
quiet for hidden panels, backlogged panels, internal `ut spawn` sessions, and Working → Waiting
transitions. If both signals arrive together, Needs You takes priority. Both alerts are off by
default and have separate duration controls.

Argus writes the keyboard's dedicated HID LED output; it does not synthesize a Caps Lock keypress
or toggle capitalization. When a burst ends, is cancelled, or the attention queue clears, the light
is restored to the current real Caps Lock state. The alerts work with built-in or external
keyboards that expose a standard Caps Lock LED element.

## How it works

Each session already has a deterministic **state dot** in the sidebar (running / waiting / idle),
read straight from the screen. The Command Center adds a layer on top of that: a small language
model that reads the *content* of the screen and writes the human line.

On macOS, Argus periodically pulls each active session's recent scrollback from its broker and asks
a fast model (`claude -p`, Haiku) for a compact `{label, summary}` judged against what you asked
that agent to do. The result is published to the owning broker so any client — your phone included —
can read it. There is no central service: the Mac generates, the brokers relay, the phone displays.

The inferred session status is a read of the **current** screen every time, so it follows reality. If an agent that
was "working" hits an API error and stops, the card flips to **stuck**; when you hand it new
instructions, it flips back to **working**. An agent minding launched work — a training run,
a sweep, a monitoring loop it set up — counts as **working** even while its own prompt sits idle,
so long jobs don't read as "idle". A status you set by hand is delivered to the model as a
correction it keeps in context, so it sticks until the screen genuinely changes — without being
permanently forced.

<Callout type="info">
The Mac generates model-written session summaries and publishes them to the brokers; Android renders
the same cards and can change manual status. Lab approval cards are deterministic protocol state,
not model inference, and both macOS and Android render and notify for them. Model-written refreshes
cost a small amount; unchanged screens are skipped, and hidden / agent sessions are never summarized.
</Callout>
