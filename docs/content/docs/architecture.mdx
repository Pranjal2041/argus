title: Architecture
description: How one client reaches every session on every machine, with no central server.

Argus is built in swappable layers. Only one of them — the **SessionProvider** — knows about
a specific multiplexer or OS; everything above it is generic.

## Core principle

> `ut` is a drop-in for `tmux`. The unit is the **tmux server (its socket)** — not a SLURM/PBS
> job — so the same binary works identically on a cluster node, a plain SSH box, or your Mac.

The first `ut` on a socket lazily starts **one broker** for that server; every later `ut`
reuses it. The broker publishes that server's sessions to your tailnet and exits when the
server dies, so its device removes itself. You isolate brokers by choosing a different socket
(`ut -L <name>`), exactly as with `tmux`.

## Discovery — capability-based, never by hostname

The client does **not** find brokers by hostname or tag. It probes port `8722` on every online
tailnet peer and accepts only those that answer the broker identity handshake:

```
GET /whoami  →  { "marker": "universal-tmux-broker", "name": "<display name>" }
```

So an unrelated service on `8722` is never mistaken for a broker, and brokers need no renaming
to be found. A `tag:utmux-broker` ACL stays **optional** — used only to gate which devices the
client may reach. Brokers join with an **ephemeral** key, so a dead one's tailnet device
auto-removes.

## Protocol

One binary **WebSocket** per client, with a 1-byte opcode framing two lanes on the same socket:

- **raw** — `OUTPUT(pane, bytes)` down; `INPUT(pane, bytes)`, `RESIZE(pane, w, h)` up.
- **control** — JSON `EVENT` (topology) and request-id `RPC` (list / rename / spawn / open-file).

Flow control uses `refresh-client -f pause-after=N` so a streaming pane can't back up and trip
tmux's slow-control-client disconnect (`%pause` / `%continue`, confirmed on tmux 3.2a).

## Agent state — a passive footer read

Each session carries a `state` the broker computes **passively**, with no input ever sent to the
agent. On Unix it reads the visible screen with `tmux capture-pane`; on Windows it reads the
session's recent ConPTY output ring (there is no `capture-pane`). It scans the last few non-blank
lines for the agent's `esc to interrupt` footer — both **Claude Code** and **Codex** print that
only while a turn is active — and reports `working` if present, else `idle`. The check is
agent-agnostic and needs no config: a noisy pane (a server logging in the same window) never prints
the footer, so output activity alone can't false-positive. The macOS client polls `/sessions` (~2s)
and renders a pulsing blue dot for `working`, a solid green one otherwise.

## Clusters are not a special case

A cluster is just a machine where you run `ut` inside an allocation. Nothing reads
`$SLURM_JOB_ID` or any scheduler variable. **Durability = the allocation, not tmux**: the
scheduler's cgroup tears down every process — broker and tmux server alike — when the job ends,
so recovery is `--requeue`-style auto-rerun plus a **stable socket name** the client transparently
reattaches to when the job comes back.

## User-data sync: your Mac is the sync host

[Workflows](/docs/workflows), [Todo Maps](/docs/todo-maps), and the [Notes Hub](/docs/notes) are
**user-global** — they belong to you, not to any one host like a session or a hidden panel does. To
keep them in step across your Mac and your phone with no central server, one broker is the **sync
host**: your **Mac's** broker. Each app keeps a local copy and reconciles with it. A small
`/userdata` key-value store on the broker holds each blob, every change carries a timestamp, and the
**newer** copy wins (last-write-wins). Your Mac talks to its own loopback broker; your phone reaches
the Mac broker over the tailnet, which it already does for sessions. So a workflow you add on the Mac
shows up on the phone within seconds, and a note you jot on the phone lands on the Mac just as fast.

<Callout type="info">
The broker, the file service, and the port-forward agent all ride the **same** per-host broker
over the tailnet — so adding Files or Ports needs zero extra deployment.
</Callout>
