A lot of what agents produce lives behind a web UI on the machine that ran them — a training
dashboard, a TensorBoard, a Jupyter server. Reaching those normally means an `ssh -L`, a forwarded
port, and a browser tab. Argus folds that into the app: a built-in, tabbed browser that reaches any
host's web service over the same tailnet, plus first-class Jupyter notebooks that run on the host.

## Dashboards

The **Dashboards** window is a small web browser with durable tabs, back/forward/reload, an address
bar, `⌘F` find-in-page, and independent `⌘+` / `⌘−` / `⌘0` zoom. Open a tab against a host
and Argus sets up the [port-forward](/docs/ports) for you, so a server bound to
`localhost:6006` on a cluster node is just *there*, no manual tunnel.

The empty window is a **service catalog** across your machines. Argus inspects listening ports and
performs an actual HTTP probe before showing a service card; it does not guess from a process or
port name. Desktop-app noise is hidden by default and available behind **Show all**.

For dashboards that should stay current without interaction, each tab can auto-refresh every 5,
15, 30, or 60 seconds. Open tabs and their host/port identity persist across app launches.

## JupyterLab

**New → JupyterLab** asks for a machine and a folder, and opens the host's **full JupyterLab rooted
at that folder** in the main pane — file browser, launcher, and all. Create or open notebooks from
the Lab itself. The key point: **the kernel runs on the host** — the GPU node, with its environment
and data — while you edit and run cells from your Mac. No SSH, no copying notebooks around, no local
kernel.

The pane is durable in both directions: switching to a terminal and back does **not** reload the Lab
(your unsaved cell edits stay put), and open Labs are remembered across app restarts — the server
keeps running on the host, and reopening reconnects to it. The broker also launches Jupyter with its
network phone-homes disabled, so a cold start on an egress-restricted cluster node takes seconds,
not minutes, and autosave is tightened to 30s so an app quit loses at most that much.

<Callout type="info">
Both ride the existing per-host broker and the port-forward hub — there is no extra service to deploy.
A notebook's kernel and a dashboard's server live on the machine that owns the work; Argus just gives
you a window onto them.
</Callout>
