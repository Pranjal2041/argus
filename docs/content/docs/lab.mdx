Argus Lab is the research layer of Argus. Agents run experiments through the `ut lab` CLI; you
approve, inspect, compare, guide, and curate them from the Lab hub on macOS or Android.

The premise is deliberately strict: an agent's memory is not a lab notebook. Code state, parameters,
data fingerprints, environment, logs, and exit status are captured by the tool at execution time.
Results remain attributable to whoever reported them, while guidance that agents should treat as
ground truth remains human-authored.

## The protocol

  <Card title="Human gates" description="A set policy decides which experiments require approval. Approved code and parameter hashes are bound to the proposal, so changed inputs require a fresh decision." />

The hierarchy is **project → experiment set → run**. A run may also carry a group label for a sweep
or ablation. The set is the isolation and access boundary; the group is an organizational label.

## Agent workflow

An agent requests a key once for the experiment set:

```sh
ut lab login
export UT_LAB_KEY=<printed-key>
ut lab brief
```

The request appears in Lab and in Command Center. Until you approve it, every protected Lab command
fails. Once approved, `ut lab brief` becomes the agent's durable briefing: applicable human
guidance, set policy, runs and their latest reported results, and pending or rejected proposals.

Plant the protocol reminder in a project's `CLAUDE.md` or `AGENTS.md` with:

```sh
ut lab init
```

Every experiment then goes through the wrapper:

```sh
ut lab run \
  --tier full \
  --group router-ablation \
  --intent "measure whether the frozen teacher improves routed accuracy" \
  --params conf/train.yaml \
  --data-files /shared/eval.jsonl \
  -- python train.py --config conf/train.yaml
```

The `--` separator is intentional: Lab flags come before it, and the exact experiment command
comes after it. Declared data paths must be absolute so Lab can fingerprint them at start and end.

Afterward, the agent appends conclusions to the run rather than leaving the only result buried in a
terminal log:

```sh
ut lab note --run R12 "Routed acc@100 improved 11.6pp; dense improved 7.2pp."
ut lab show R12
ut lab diff R12
```

`show` reads the recorded run. `diff` compares the recorded base commit, parameter files, and
declared data hashes with their current state, and calls out when the run carried an uncommitted
snapshot—the useful first question when an old result no longer makes sense. The full recorded
environment remains visible in `show` and the hub.

## Approval policies

Each set has one policy:

| policy | behavior |
|---|---|
| `full-only` | Quick probes start immediately; full experiments need approval. This is the default. |
| `all` | Every wrapped run needs approval. |
| `none` | Runs start immediately, but are still fully recorded. |

A gated command files a proposal containing its intent and captured inputs, then waits for a
decision. Approval launches it automatically while the command is waiting. If the wait expires, the
proposal stays pending and can be launched later:

```sh
ut lab run --proposal R12
```

Launch recomputes the code and parameter hashes. If they differ from the approved proposal, Lab
refuses to run it and asks for a new proposal. A rejection can carry a human explanation which the
agent receives in its brief.

### Unattended Mode

Turn on **Unattended Mode** when you want Lab work to continue while you are away. The Mac broker
then approves new Lab access-key requests and recorded run proposals automatically, even when the
Argus app and Lab pane are closed. You can control the same broker-owned switch from Lab, Settings,
or Command Center on macOS, and from the drawer or Command Center on Android.

This is explicit user pre-authorization, not a weaker run path. Every decision is recorded with the
note `Auto-approved because Unattended Mode was on.`, and approved proposals still undergo the same
code and parameter bind-hash check before launch. Turning the mode off stops future automatic
decisions; approvals already committed remain part of the append-only record. Keep the Mac awake
and reachable if it needs to approve requests from remote machines.

The current scope is intentionally limited to Lab gates. Unattended Mode does not yet answer
free-form questions in terminal agents.

## The Lab hub

Open Lab with `⇧⌘L` on macOS or the Lab destination on Android.

- **Inbox** contains pending access requests and run proposals, newest first. Each proposal opens an
  evidence dossier before the approve/reject controls.
- **Research** groups experiment sets by project and opens every run's reported results, parameters,
  exact code snapshot and diff, declared-data integrity, environment, log, artifacts, and event
  record. Agent-authored Markdown, tables, and code blocks render as rich, safe content.
- **Access** in the macOS dashboard is the equivalent of `ut lab keys`: it lists active, pending, denied, and
  revoked credentials without exposing full secrets. Pending requests open their decision dossier;
  active keys can be revoked, and assigned sets open directly from the registry.
- **Compare** selects two runs from a set and shows their recorded envelopes and parameter delta.
- **Guidance** publishes durable human instructions everywhere, to one machine, to one project, or
  to one experiment set. A set ledger shows both its direct instruction and inherited guidance.
- **Curation** hides individual claims, archives and restores runs or sets, changes approval policy,
  and revokes keys without deleting the underlying record. A running record whose real process is
  already gone can be marked stopped with an audited reason.

The set page keeps the complete human control surface visible: archive or restore the set, change
its approval policy, revoke access, and install the standard Lab instruction into that project's
`AGENTS.md` / `CLAUDE.md` files (the same idempotent operation as `ut lab init`). Access-request
approval also accepts the initial policy and the optional human decision note supported by the CLI.

### Stopping an orphaned run record

Normally `ut lab run` writes a mechanical ending with the process exit code. If the wrapper itself
disappears—for example during a node reset—but you later verify that no process or scheduler job is
still running, the run would otherwise remain “Running” forever. Open that run and choose **Mark
stopped**. Both macOS and Android require:

1. a reason explaining why the automatic ending is missing; and
2. confirmation that you independently checked the underlying process or job.

This writes a human-authored `run-stop` event and gives the run the neutral **Stopped** phase. It
does not send a signal and does not invent an exit code, so Stopped is neither Finished nor Failed.
If the original wrapper later reports a genuine `run-end`, that mechanical outcome supersedes the
manual correction. The CLI equivalent is:

```sh
ut lab mark-stopped <set> <run> --reason "wrapper vanished; scheduler confirms no surviving job"
```

Archive is separate. Archiving only removes a set or run from normal views, is reversible, and does
not change lifecycle state or affect a process. Consequently, archiving a Running record does not
make it Stopped; use **Mark stopped** first when the process is truly gone, then archive it if you no
longer want it in the normal ledger.

Lab attention is not confined to the hub. Pending access and experiment decisions appear in
**Command Center** beside agent sessions, and system notifications deep-link to the exact dossier.
This works on macOS and Android. Command Center also exposes the Unattended Mode switch and shows
its active state, so the automation is visible from the surface you monitor.

## Shared clusters and offline records

On an NFS-backed cluster such as a fleet of `babel-*` nodes, every node can see the same Lab store.
Argus uses the broker-reported store identity—and a cluster-prefix fallback—to show one experiment,
key, proposal, or instruction rather than one replica per node. Actions still route to an online
owner.

The Mac broker also mirrors remote Lab records. When a source machine goes offline, its mirrored
sets remain readable and are marked as offline; actions that require the source are disabled. The
record is append-only and the mirror never silently shrinks.

## What the record can and cannot promise

Lab promises a complete record of work that passed through `ut lab run`. It cannot stop an agent
from bypassing the wrapper, and data provenance covers only files declared with `--data-files`.
The store is owned by the same operating-system user as the agent, so append-only is a protocol
guarantee at the CLI and UI surfaces rather than a hostile-process security boundary.

The practical contract is simple: a research result only counts when it has a Lab run ID.

## Human-side CLI

Most human work is easier in the hub, but the CLI has equivalents:

```sh
ut lab keys
ut lab approve <key-prefix> --project my-project --policy full-only
ut lab deny <key-prefix> --note "use the shared eval split"
ut lab proposals
ut lab grant <set> <run>
ut lab reject <set> <run> --note "increase the sample size first"
ut lab policy <set> all
ut lab mark-stopped <set> <run> --reason "why the wrapper did not record an ending"
ut lab archive <set> [run]
```

Run `ut lab help` for the complete agent and human command reference.
