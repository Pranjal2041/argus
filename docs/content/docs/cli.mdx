description: A drop-in for tmux that publishes a host's tmux server to your tailnet.

`ut` is a thin wrapper around `tmux`. It behaves like `tmux` for attaching and creating
sessions, and additionally makes sure exactly **one broker** is running for the chosen tmux
socket so the server is reachable from the Argus app.

## Usage

```sh
ut [name]               # attach or create a session (default name = current dir)
ut -L <socket> [name]   # use a separate tmux server + broker, like `tmux -L`
```

- A **named** argument reconnects (attach-or-create), exactly like `tmux new-session -A`.
- **No** name makes a new session every time, named after the directory with a numeric suffix
  when the name is taken — so you can run several sessions from one directory.

## The model

The first `ut` on a socket lazily starts one broker for that server; later invocations reuse it.
The broker joins the tailnet via `tsnet` using a single shared key
(`~/.universal-tmux/authkey`) but keeps its device **identity** node-local — so dozens of jobs
across nodes each become their own tailnet device off that one key and never clash. With no key,
the broker serves on loopback (useful on a Mac that's already on the tailnet).

The client discovers brokers by the [`/whoami` handshake](/docs/architecture#discovery--capability-based-never-by-hostname),
so device names don't matter to it.

## The mesh fabric

Beyond attaching, `ut` is also a small **cross-host fabric** — the primitives you (or an agent) use
to reach and orchestrate work on any machine on the tailnet, by name, without SSH. Each runs through
the target host's broker.

| command | what it does |
|---|---|
| `ut ls` | list every session across all reachable hosts |
| `ut exec <@host[:shell]> <cmd…>` | run a command on a host and return its output |
| `ut sh <@host> [name]` | create a persistent **agent shell** on a host (keeps env / cwd / venv; hidden by default) |
| `ut sh --visible <@host> <name>` | explicitly create a persistent user-visible shell |
| `ut run <host:shell> <cmd…>` | run a command inside one of those persistent shells |
| `ut spawn [--idle <dur>] <@host[:name]> <cmd…>` | fire a long job into a fresh session |
| `ut send <@host:shell> <text…>` | type text into a session |
| `ut tail <@host:session>` | read a session's recent output |
| `ut cp <src> <dst>` | copy a file (either side may be `host:path`) |

Sessions created by the mesh are **agent sessions**, so agents can orchestrate freely without filling
the main sidebar:

- `ut sh` creates a stateful agent shell, hidden by default and removed after seven days without
  activity. Its cwd, exports, and activated environment still persist across `ut run` calls.
- `ut spawn` creates a background job session, hidden by default and **auto-reaped** once the finished
  job has sat idle past its leash (`--idle`, a few hours by default). A live command is not removed.
  After completion, its retained terminal is removed after at most seven days even when `--idle 0`
  disables the shorter cleanup.

Use `ut sh --visible` only when the persistent shell is deliberately part of the user's normal panel
workspace. Sessions created with the Argus **New Session** control and interactive `ut <name>` remain
user-visible.

## The Lab research protocol

`ut lab` is a second command family implemented by the same CLI. It does not create or attach a
terminal session; it reads and writes the append-only [Argus Lab](/docs/lab) store:

| command | what it does |
|---|---|
| `ut lab login` | request an approved key for one isolated experiment set |
| `ut lab brief` | read applicable human guidance, run summaries, and pending decisions |
| `ut lab run … -- <command>` | capture provenance, apply the set's approval policy, and execute |
| `ut lab note --run R3 <text>` | append an attributable result or observation |
| `ut lab show R3` | inspect one run's full record |
| `ut lab diff R3` | compare the recorded base commit, parameters, and declared data with today |
| `ut lab init` | add the durable Lab reminder to `CLAUDE.md` or `AGENTS.md` |

The human approval, policy, curation, archive, and key commands are documented in the
[Lab guide](/docs/lab). Run `ut lab help` for the complete terminal reference.

## Environment

| variable | default | meaning |
|---|---|---|
| `UT_HOME` | `$HOME/.universal-tmux` | shared dir (NFS ok): the broker binary + the one auth key |
| `UT_BROKER` | `$UT_HOME/ut-broker` | path to the broker binary |
| `UT_PORT` | `8722` | port the broker listens on |
| `UT_LOCAL_DIR` | `/tmp/ut-$USER` | **node-local** state: identity, pidfiles, logs (never NFS) |

<Callout type="info">
Isolation between two allocations on one node is opt-in: each passes a distinct socket
(`ut -L <name>`), exactly as with `tmux`. By default they share the node's default server and
one broker.
</Callout>

## tmux version notes

- Control mode works on **tmux 3.2a** (subscriptions and `pause-after` confirmed).
- **OSC 8 ⌘-click** to open files needs **tmux ≥ 3.4** — build 3.5a/3.6 into `$HOME/local`
  for fresh sessions; never `kill-server` a shared one.
